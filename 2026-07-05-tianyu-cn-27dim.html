D0 — 赛道真实性验证

验证项 结论
是否属于 HaH（Hospital-at-Home）？ 🚨 否——天与养老是数字化居家养老服务平台+B2G 运营服务商，主要提供生活照料、基础护理、长护险服务和社区养老服务，不提供急性期 HaH 或临床诊疗服务。其服务属于「长期照护（LTC）」范畴，非急性期医疗。
对公司类型判定 🏢 私有公司（上海天与养老服务有限公司），2019 年正式注册运营
对 iHomeCare 相关性评估 🟡 中高——天与养老的核心模式「技术平台 × 线下运营」对中国养老市场有参考价值；其 SaaS 平台架构、IoT 养老硬件部署、AI 失能评估等技术路径值得 iHomeCare 借鉴，但 B2G+长护险混合模式高度依赖中国政策环境，不能直接跨境复制。
赛道本质 中国「科技养老」赛道——既不是纯 SaaS 技术公司（如美国 AlayaCare），也不是纯服务运营商（如福寿康），而是介于两者之间的混合体。天与养老代表了中国特色养老赛道中的一个独特定位：「技术驱动的养老服务运营商」。
赛道成熟度 中国居家养老市场处于早期快速扩张阶段——政策驱动（长护险试点+政府购买服务）是核心增长动力，市场集中度低，头部公司市占率不足 5%。天与养老处于该赛道的头部梯队，但与福寿康、小橙长护等仍处于「跑马圈地」阶段。
政策敏感性 🚨 极高——该赛道的核心驱动力是政策（长护险全国推广节奏、政府购买养老服务预算、各地养老数字化监管要求），政策波动可能直接改变赛道格局。
赛道价值链位置 天与养老占据「运营服务」和「技术平台」的交叉点——上游是 IoT 硬件厂商（鱼跃、华为等）和护理员劳动力市场；下游是政府端（民政/医保招投标）和 C 端老年消费者。在中游，天与同时扮演「服务集成商」和「技术输出方」双重角色。
赛道结构性吸引力 🟡 中等——刚需明确（老龄化加速）、政策支持（十四五规划重点）、增长空间大，但进入壁垒不高（技术门槛有限）、利润率偏低（B2G 和长护险利润薄）、对政策高度依赖。头部公司靠规模效应和 G 端关系建立护城河。

核心判断（总分论点）：

  1. 赛道判断 🟢：中国科技养老赛道确定性高——老龄化不可逆（3.1 亿 60+ 人口，2024 年数据），政策持续加码（长护险 49 城市试点→全国推广），天与养老占据有利位置。

  2. 模式判断 🟡：混合模式（技术+运营）在中国养老市场可能是最优解——纯 SaaS 在中国 B 端付费意愿低，纯运营利润薄且难以规模复制。但「双线作战」对管理团队的要求极高——需要在「技术效率」和「服务质量」之间找到平衡。

  3. 竞争判断 🟡：天与养老在「技术」维度领先竞品（福寿康/小橙长护），但在「运营深度」和「品牌认知」维度尚未建立绝对优势。长期竞争格局可能演变为「天与 = 技术平台型 vs 福寿康 = 服务品牌型 vs 小橙长护 = 长护险深耕型」三足鼎立。

  4. 财务判断 ⚠️：推测 2024 年收入 ¥4–8 亿（高速增长期），但盈利压力存在——B2G 项目利润率低（5–10%），长护险服务净利率 3–8%，SaaS 高毛利但占比小。整体可能接近盈亏平衡或微利。

  5. 风险判断 🔴：最大风险是政策不确定性——长护险全国推广节奏直接影响天与的核心收入增长；其次是政府预算收紧影响 B2G 业务持续性。这两个风险短期内无法通过公司自身策略消除。

  6. 估值判断 🟡：推测估值区间 $150–400M（约 ¥10–30 亿），对应收入倍数约 3–8x。技术溢价使估值高于纯运营商（福寿康可能 2–5x），但低于纯 SaaS 公司（10x+）。高瓴/长岭/钟鼎/大华创投等头部资本的参与提供了估值底部支撑。

  7. 对 iHomeCare 启示 🟡：天与养老的「SaaS 平台 + IoT 硬件 + AI 能力」三件套技术路径可借鉴；其「以政府需求驱动产品设计」的思路值得参考；但 B2G 获客模式难以跨境复制。

D3 — 战略定位（重点关注）

3.1 核心战略命题

天与养老的战略定位是中国养老行业中「技术派」的代表。其核心战略判断可拆解为四个层次：

层次一：中国养老 ≠ 纯技术市场

  • 中国养老行业的付费主体是政府（民政/医保），而非 C 端消费者或 B 端机构
  • 纯 SaaS 在中国养老 B 端付费意愿极低——中小养老机构年 IT 预算普遍 ¥1 万以下，无法支撑纯软件公司盈利
  • 因此必须捆绑服务运营，「技术」是手段而非目的

层次二：技术作为「杠杆」

  • 用 SaaS 平台提升服务管理的标准化和可追溯性（解决养老服务质量管控难题）
  • 用 IoT 设备实现 7×24 安全监测（解决独居老人安全问题）
  • 用 AI 算法优化护理员调度（解决人员效率问题）
  • 核心逻辑：技术投入 → 运营效率提升 → 规模扩张 → 单位成本下降

层次三：B2G 作为「切入点」

  • 政府购买居家养老服务（各地民政局招标）和长护险（医保支付）是当前最稳定的收入来源
  • 技术能力是拿政府订单的差异化武器——帮助政府实现「养老服务的数字化监管」
  • 技术+服务双背书使天与在政府招投标中区别于纯运营商

层次四：B2C 作为「终局」

  • 长期目标是建立 C 端品牌，直接面向老年消费者和家属收费
  • 路径：G 端获客 → 服务留客 → 口碑传播 → C 端复购 → 增值服务变现（如适老化改造、智能硬件的 C 端销售）
  • 但 C 端消费习惯在中国老年群体中尚未成熟，转化周期长

3.2 战略口号与品牌定位

  • 品牌口号：「用科技让长者生活更美好」（推测自行业报道）
  • 品牌形象：技术驱动、专业可靠、政府背书
  • 品牌资产：高瓴创投/长岭资本等顶级基金背书是重要的信任来源

D4 — 商业模式（重点关注）

4.1 混合收入模式总览

天与养老的核心创新不在技术本身，而在于「多重收入引擎」的商业模式设计——同时跑通 B2G、长护险、SaaS、B2C 四条收入线，使其在收入结构上比纯运营商（福寿康）更抗风险，比纯技术公司（纯 SaaS 创业公司）更务实。

4.2 B2G 政府购买服务（核心收入）

模式说明：

  • 各地民政局/老龄办通过招投标采购居家养老运营服务
  • 天与养老作为中标方，运营社区养老服务中心/驿站，提供上门服务
  • 结算方式：按服务人头/按项目/按服务次数结算
  • 合约周期：通常 1–3 年，到期续约

典型项目（推定）：

  • 社区养老服务中心（日间照料中心）运营
  • 政府购买居家上门服务（为特定老人群体提供免费或补贴服务）
  • 适老化改造项目（政府出资为困难老人家庭进行安全改造）
  • 智慧养老平台建设和运维（政府数字化监管平台）

B2G 业务的利润结构：

  • 收入端：政府购买服务单价通常 ¥500–2,000/人/月（按老人照护等级和各地经济水平不同）
  • 成本端：护理员工资（占比 60–70%）、管理人员成本、平台运维、场地费用
  • 净利润率：推测 5–10%（行业典型水平，利润空间受制于政府预算约束）
  • 优势：收入可预测、客户粘性高（政府关系建立后替代成本高）
  • 劣势：回款周期长（3–6 个月常见）、政策波动风险、各地标准不统一

4.3 长护险服务（增长引擎）

模式说明：

  • 天与养老在各地申请成为长护险定点护理服务机构
  • 经评估确认为失能/半失能的老人，可享受长护险支付的上门护理服务
  • 服务内容和价格由各地医保局统一规定
  • 结算方式：医保/社保基金直接拨付至机构（部分城市需个人先垫付再报销）

长护险单位经济模型（典型城市）： | 项目 | 数据（估） | |------|-----------| | 单次上门服务时长 | 1–2 小时（常见 1 小时） | | 单次服务收费 | ¥40–70/小时（各地差异大，上海约 ¥50–55，成都约 ¥40–45） | | 护理员时薪 | ¥25–35/小时（含社保雇主部分） | | 平台抽成/管理费 | ¥5–12/小时 | | 单次服务毛利润 | ¥5–15/小时 | | 单个护理员日均服务量 | 4–6 单/天 | | 单个护理员月产值 | ¥5,000–10,000/月（按 22 天计） | | 长护险客户 ARPU | ¥500–2,000/月（取决于照护等级和城市） |

关键成功要素：

  • 成为各地长护险定点机构的审批门槛（需满足人员、场地、系统等要求）
  • 服务质量管理（长护险有严格的稽核和考核机制）
  • 护理员招聘和留存（行业普遍难题）
  • 与医保局的结算系统对接（SaaS 平台需要支持各地不同的结算口径）

4.4 SaaS 平台订阅（高毛利业务）

模式说明：

  • 面向中小养老机构/社区养老服务商的数字化管理工具
  • 核心功能：客户管理（CRM）、排班调度、服务工单、政府监管数据对接、财务报表
  • 定价模式：年费制

SaaS 定价推测： | 机构规模 | 年费（推测） | 功能范围 | |----------|------------|---------| | 小型（<20 张床位） | ¥3,000–8,000/年 | 基础 CRM + 排班 | | 中型（20–100 张） | ¥10,000–30,000/年 | CRM + 排班 + 工单 + 财务 | | 大型（100+ 张） | ¥30,000–100,000/年 | 全功能 + 政府数据对接 + BI | | 集团版（多机构） | ¥100,000–300,000/年 | 多租户 + 定制开发 + 运维支持 |

SaaS 业务的核心价值主张：

  1. 帮助机构满足政府监管要求——各地民政越来越要求养老服务机构上线数字化监管系统，天与的 SaaS 天然对接各地监管平台
  2. 提升运营效率——智能排班、自动工单、数据报表
  3. 降低合规风险——服务全流程留痕，应对政府稽核
  4. 数据驱动决策——BI 报表辅助管理决策

SaaS 业务的挑战：

  • 中国养老机构普遍数字化意识薄弱，SaaS 渗透率低
  • 中小机构付费意愿有限（年费门槛对他们来说是显著成本）
  • 竞品低价竞争（部分本地 IT 公司以 ¥1,000–3,000/年抢单）
  • 续约率是关键指标（推测 70–85%）

4.5 B2C 自费服务（长期潜力）

模式说明：

  • 直接面向老年消费者和家属的高附加值服务
  • 与 B2G 和长护险的「补贴/政府支付」服务形成差异化

典型自费服务： | 服务 | 定价（推测） | 竞争壁垒 | |------|------------|---------| | 适老化改造（居家安全） | ¥3,000–20,000/户（按改造范围） | 设计能力+施工团队+政府推荐 | | 智能设备零售 | ¥100–1,000/件 | 性价比+售后服务 | | 自费上门护理（非长护险覆盖） | ¥60–150/小时 | 服务质量+品牌信任 | | 健康管理增值服务 | ¥200–500/月 | AI 健康监测+人工关怀 | | 养老规划咨询 | ¥500–2,000/次 | 专业能力+信任基础 |

4.6 收入结构的战略意义

当前结构（推测）：B2G + 长护险 ≈ 70–80%（政策驱动） 理想结构（3–5 年后）：SaaS + B2C ≈ 30–50%（市场驱动）

战略核心目标：逐步降低对政策驱动收入的依赖，提升 SaaS 和 B2C 的市场驱动收入占比。这是一个长期转型过程，需要 C 端品牌建设和 SaaS 市场渗透率提升的双重支撑。

5.6 天与「无为算法系统」（2025.4 发布）[Tavily-36氪/中国日报]

产品概述： 2025年4月，天与养老正式发布 「无为算法系统」——国内首个基于AI驱动的毫米波雷达居家场景解决方案。该产品是继SaaS平台、IoT硬件之后，天与在AI感知技术领域的重要突破。

技术原理： | 技术要素 | 说明 | |---------|------| | 感知硬件 | 毫米波雷达（非摄像头，不侵犯隐私） | | AI算法 | 深度学习模型，可识别跌倒、滞留、长时间不动、异常行为等 | | 部署方式 | 壁挂式，覆盖单房间（客厅/卧室/卫生间） | | 数据闭环 | 雷达数据 → 边缘AI推理 → 异常告警 → 服务工单 → 人工响应 | | 隐私保护 | 不采集图像/声音，仅获取点云位置数据，完全符合隐私法规 |

应用场景：

  1. 跌倒实时检测：老人跌倒后自动报警，响应时间 <10秒
  2. 行为异常监测：长时间未活动/卫生间滞留过久 → 自动提醒
  3. 睡眠质量分析：夜间睡眠时长、翻身频次、离床次数
  4. 活动轨迹分析：日常活动模式 → 异常偏离自动预警

战略意义：

  • 从「被动响应」到「主动感知」——不再等待老人按呼叫器，而是系统主动发现异常
  • 毫米波雷达 vs 摄像头：隐私保护是居家养老场景的核心需求，毫米波雷达无图像采集，更容易被C端用户接受
  • 与长护险服务形成协同：智能监测减少上门频次（降低运营成本），同时提升安全覆盖率

市场反响： 「无为算法系统」发布后获得行业高度关注，入选国家三部委推广目录，并获2024年度数智化创新成果奖 [Tavily-36氪/中国日报]。该系统是天与从「服务数字化」向「AI主动感知」转型的标志性产品。

9.2 SaaS 平台深度分析

前端技术栈（推测）：

  • Web 端：React / Vue.js（中国主流前端框架）
  • 移动端：微信小程序（护理员端）+ 企业微信（管理端）
  • 适配性：支持手机、平板、PC 多端

后端技术栈（推测）：

  • 后端语言：Java（Spring Boot）或 Go（国内企业级 SaaS 常见选择）
  • 数据库：MySQL / PostgreSQL + Redis 缓存 + MongoDB（IoT 数据）
  • 云平台：腾讯云（行业常用选择，与微信生态协同）
  • 微服务架构（支撑多城市多租户）

核心技术特点：

  1. 多租户架构（Multi-tenant） - 支持不同城市、不同机构在同一平台独立运营 - 数据隔离（每个机构的数据严格隔离） - 功能可配置（不同机构的业务规则可定制）

  3. 离线能力 - 护理员端支持离线签到和记录（网络恢复后自动同步） - 边缘网关在网络中断时缓存 IoT 数据

  4. 安全合规 - 数据加密存储和传输（国密算法支持） - 等保三级（等保 2.0 三级要求） - 个保法合规（老年人健康数据特殊保护）

16.3 需谨慎之处

不宜跨境复制：

  • B2G 模式高度依赖中国政府购买服务体系和政府关系网络
  • 长护险制度是中国特有的社会保障制度，其他市场无直接对标
  • 「低价竞标政府项目」策略在利润驱动的海外市场可能不适用
  • 护理员薪酬结构（低底薪+按单提成）在中国以外市场可能面临劳动法规挑战

需要适配：

  • SaaS 平台功能需要适配海外养老/医疗支付体系（保险/Medicare/Medicaid/NHS 等）
  • IoT 硬件需要符合海外安全认证（CE/FDA/RoHS 等）
  • AI 模型需要基于海外老人数据重新训练（人群差异+文化差异）

  1. 技术平台建设：采用类似天与的「SaaS 核心 + IoT 扩展 + AI 智能」三步走路径

  3. 产品设计：以「满足合规要求」为第一优先级（天与 SaaS 的核心价值）
  4. 离线能力：优先保障核心功能在弱网/离线环境的可用性

  • 49 个试点城市，覆盖约 1.8 亿人口
  • 基金收入约 ¥300–500 亿/年（2024 年估）

  • 筹资模式：医保基金划转+个人缴费+财政补贴（各地差异大）
  • 服务内容：以居家照护为主（占比 70–80%），辅以机构照护和社区照护

全国推广的关键议题： | 议题 | 当前状态 | 对天与的影响 | |------|---------|------------| | 筹资来源 | 医保基金可持续性争议 | 若筹资不足，可能降低服务标准（影响收入） | | 覆盖范围 | 是否包含轻度失能 | 扩大覆盖 → 客户池扩大 | | 服务标准 | 全国统一 vs 地方差异 | 统一标准有利于规范化，但可能降低地方灵活性 | | 支付标准 | 各地差异大 | 全国统一支付标准可能利好/不利不同城市 | | 护理员资质 | 是否要求持证 | 持证要求提高行业门槛，对规范化运营的天与有利 |

  • 乐观情景（概率 25%）：2026 年加速推广，覆盖范围扩大
  • 悲观情景（概率 25%）：受医保基金压力影响，推广节奏放缓
