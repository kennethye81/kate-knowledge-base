维度 1 — 执行摘要

一句话定位：Medically Home 是美国医院居家医疗（Hospital at Home）赛道的技术基础设施”军火商”——为 Mayo Clinic、Kaiser Permanente 等顶级医疗系统提供从 Virtual Command Center、临床协议到供应链的全栈 HaH 技术平台，2025年因现金流枯竭被迫与 DispatchHealth 合并。

核心矛盾：顶级医疗机构背书（Mayo Clinic、Kaiser Permanente 联合投资约$100M）+ 超大 TAM（美国住院市场>$1万亿） vs. 独立商业模式无法持续（烧光$287M融资后估值大幅缩水、员工普通股清零、被迫卖身 DispatchHealth）。

关键数据速览 数值
成立时间 2016年
总部 Boston, MA
累计融资 ~$287M（PitchBook）/ $197M（Tracxn，口径不同）
核心投资者 Mayo Clinic、Kaiser Permanente、Baxter、Cardinal Health、GMR
服务患者数 >40,000（截至2024年8月）
2025年状态 已被 DispatchHealth 合并，品牌消失
估值轨迹 2022年峰值→2025年”大幅缩水”，员工普通股清零

总体判断：Medically Home 是 HaH 赛道最具战略眼光的早期玩家，但商业模式上的结构性缺陷（高固定成本虚拟指挥中心 + 依赖少数大客户 + CMS Waiver 政策不确定性）导致其无法独立生存。2025年与 DispatchHealth 的合并是”自救式”退出，而非战略升级。对 iHomeCare 而言，Medically Home 的经验教训比其技术本身更有价值。

维度 3 — 战略定位

一句话核心逻辑

Medically Home 押注的是： 美国 $1万亿+ 的住院市场将被”去中心化”——通过在患者家中设置”虚拟医院病房”（技术+人员+供应链），将急性住院从高成本的医院大楼转移到低成本的患者家中，而 Medically Home 作为技术平台”军火商”赚取 SaaS/服务费。

战略差异化要素表

维度 Medically Home 定位 差异化特征
服务模式 技术赋能平台 + 临床指挥中心（Virtual Command Center） 不直接提供临床服务，而是为医院/医生集团提供”即插即用”的 HaH 运营系统
客户类型 大型医疗系统（Mayo、Kaiser等）+ 医生集团 以顶级品牌客户建立壁垒
技术核心 CESL 平台（Clinical Escalation & Support Layer） 全天候 24/7 虚拟指挥中心 + 标准化临床协议 + 供应链整合
收入模式 技术平台许可费 + 每患者服务费 混合 SaaS + 按量收费

战略演变三阶段

  • Phase 1 (2016–2019)：创立与概念验证——搭建 CESL 平台，验证急性居家医疗的技术可行性
  • Phase 2 (2020–2022)：旗舰客户获取——借助 COVID 契机拿下 Mayo Clinic/Kaiser Permanente 投资与合作，获得顶级品牌背书
  • Phase 3 (2023–2025)：挣扎求生——CMS Waiver 延期不确定性 + 高昂运营成本 + 客户扩展速度不及预期 → 现金流枯竭 → 被迫与 DispatchHealth 合并

  • 商业模式验证不足：累计融资 ~$287M 但年收入可能仅 ~$7.8M（Prospeo 2024年估），收入/融资比极低——典型的”融资驱动增长、收入模式未跑通”

核心财务矛盾

  1. 高固定成本结构：24/7 Virtual Command Center 需要大量临床人员（医生、护士、护理协调员），人力成本极高
  2. 低边际收入：按患者收费的定价受限于 CMS DRG 费率，毛利率空间有限
  3. 客户集中度风险：两大客户 = 两大投资者，几乎没有价格谈判空间
  4. 融资难以为继：2022年 Series D $110M 后再无新融资，2025年依靠投资人过桥资金苟活

维度 8 — 工作流/旅程映射

Phase 0: 需求触发

  • 患者在急诊科/门诊被诊断为需要住院但符合 HaH 条件
  • 临床评估工具判断患者适合居家治疗

Phase 1: 评估与匹配

  • Medically Home 平台接收患者信息
  • 自动匹配临床协议（心衰/COPD/COVID/感染等）
  • 确认地理位置和供应链覆盖

Phase 2: 服务交付

  • 设备+耗材配送至患者家中（通过 Cardinal Health/GMR 等伙伴）
  • 远程监控设备安装（血压/血氧/体温等）
  • 24/7 Virtual Command Center 激活
  • 每日虚拟医生查房 + 必要时上门医护/急救人员访问

Phase 3: 持续监控

  • 实时生命体征监控 + AI/规则引擎预警
  • Virtual Command Center 协调升级响应
  • 必要时 GMR 急救人员上门或转回实体医院

Phase 4: 出院/过渡

  • 达到临床出院标准
  • 过渡到康复护理或家庭医生随访
  • 设备回收

Phase 5: 结算

  • 按 CMS DRG 费率或协商费率向支付方结算
  • Medically Home 向医院收取平台服务费

盈利路径分析

核心问题：指挥中心的人力成本是固定成本——无论服务 100 还是 1,000 患者，都需要维持最低人员配置。这导致：

  1. 规模经济悖论：规模越大→需要越多指挥中心人员→固定成本同步增长→难以盈利
  2. 定价天花板：CMS DRG 费率固定，医院支付 Medically Home 的费用空间有限
  3. 盈利未验证：经过 $287M 融资后仍需过桥资金维持，商业模式盈利能力未获证实

综合判断：独立商业模式不可持续。与 DispatchHealth 合并后，通过整合临床服务团队和指挥中心实现协同效应，可能改善单位经济。

维度 16 — 战略建议（分角色）

1. 对潜在投资者

  • 估值判断：Medically Home 作为独立实体已无投资机会，已并入 DispatchHealth。合并后的 DispatchHealth 估值需重新评估。
  • 核心指标关注：合并后单位经济是否改善（指挥中心人员/患者比）、Mayo/Kaiser 客户留存率、CMS Waiver 立法进展
  • 催化剂：CMS 永久立法、《医院居家法案》通过、合并后 EBITDA 转正
  • 下行风险：整合失败、关键客户流失、Waiver 到期

2. 对 DispatchHealth 管理层（合并后）

  1. 首要任务：留住 Mayo Clinic 和 Kaiser Permanente——这两家同时是客户和投资者，任何一方流失将造成连锁反应
  2. 解决指挥中心人力成本结构性问题——利用 DispatchHealth 现有临床网络减少人员冗余
  3. 加速平台整合——将 Medically Home 的 CESL 技术与 DispatchHealth 的全美服务网络深度融合
  4. 推动 CMS Waiver 永久化立法——这是整个行业的生存基础
  5. 为2027-2028年 IPO 做准备——合并后若能实现 EBIDTA 转正，规模化故事可能支撑上市

3. 对竞对/新进入者（含 iHomeCare）

  • Medically Home 的教训比其技术更有价值：
  • ❌ 不要押注”融资驱动增长”——收入必须跑在融资前面
  • ❌ 不要让客户=投资者——这彻底消灭了议价能力和退出选项
  • ❌ 指挥中心的人力成本无法通过规模效应摊薄——这是商业模式的结构性缺陷
  • ✅ 寻找轻资产路径——做真正的技术平台（SaaS），而非半服务半平台
  • ✅ 分散客户集中度——任何单一客户占比不超过 20%
  • 差异化切入点：
  • AI 驱动的自动化指挥中心（减少人力依赖）
  • 垂直整合硬件（自研监测设备，而非外购）

GTM 阶段诊断：Product-Market Fit 前/初期规模化——技术验证完成（Mayo/Kaiser 在使用），但盈利模型未跑通，客户获取速度和经济效益未达规模化门槛。

GTM 瓶颈判断：漏斗底部（收入转化/盈利）——不是 leads 不够，而是客单价天花板低、单位经济无法支撑高 CAC。

🚨 关键发现：Atrium Health 与 Medically Home 关系修正

原假设：Atrium Health 的 HaH 项目使用 Medically Home 平台
实际验证：❌ 错误。

Atrium Health 的 Hospital at Home 项目是自建项目，始发于2020年3月 COVID 疫情期间：

  • 2021年3月获 CMS Acute Care at Home Waiver
  • 技术监控合作伙伴为 Best Buy Health / Current Health（2023年3月宣布合作）
  • Atrium 自行运营临床指挥中心和社区急救人员网络
  • 2024年预算 ADC=100，月均服务200+患者，累计服务10,574+患者
  • 是全美最大的自营 HaH 项目

对报告的影响：Medically Home 的核心客户实为 Mayo Clinic 和 Kaiser Permanente，而非 Atrium Health。Atrium 的例子恰恰说明——大型医疗系统有能力自建 HaH，不一定需要 Medically Home 的平台。
